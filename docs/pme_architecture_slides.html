Slide 2

PMEとは何か？

本質的な役割を理解する

一言で言うと

「BlenderのUIを動的に生成・登録するフレームワーク」

重要な認識: PMEは単なる「パイメニューエディター」ではない。
  • パイメニュー、通常メニュー、ポップアップ、パネル
  • マクロオペレーター、モーダルオペレーター
  • スティッキーキー、スタックキー
  • カスタムプロパティ
これら全てを統一的なデータモデルで管理し、 ホットキーと連動させて動的に登録する。

PMEが解決している問題

問題 PMEの解決策
Blender APIでUI作成は面倒 GUIエディターで視覚的に作成
ホットキー管理が複雑 メニューとホットキーを一体管理
Pythonコードが必要 ドラッグ&ドロップでコマンド追加
設定の保存・共有が難しい JSON形式でエクスポート/インポート

PMEのメンタルモデル

設計者の「世界観」を理解する

中心的な概念: 「全てはメニューである」

PMEでは、パイメニューもスタックキーもマクロも、全て PMItem という 同じデータ構造で表現されます。これが良くも悪くもPMEの特徴です。

┌─────────────────────────────────────────────────────────────────┐ │ PMEPreferences │ │ (AddonPreferences - Blenderに永続化される) │ │ │ │ ┌─────────────────────────────────────────────────────────┐ │ │ │ pie_menus: Collection[PMItem] │ │ │ │ │ │ │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ │ │ PMItem │ │ PMItem │ │ PMItem │ │ PMItem │ │ │ │ │ │ mode= │ │ mode= │ │ mode= │ │ mode= │ │ │ │ │ │ PMENU │ │ RMENU │ │ MACRO │ │ STICKY │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ pmis: │ │ pmis: │ │ pmis: │ │ pmis: │ │ │ │ │ │ [8+2個] │ │ [可変] │ │ [可変] │ │ [2個] │ │ │ │ │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ │ │ │ └─────────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────────┘
設計上のトレードオフ:
この「統一モデル」は柔軟性をもたらすが、同時に複雑さも生む。
例: パイメニューは固定8+2スロット、通常メニューは可変。
同じ PMItem で表現するため、エディターごとに解釈が異なる。
Slide 4

データモデル概要

types.py の核心を理解する

3つの主要なPropertyGroup

PMItem (メニュー定義)

class PMItem(PropertyGroup):
    name: str        # メニュー名 (ユニーク)
    mode: str        # PMENU, RMENU, MACRO...
    km_name: str     # キーマップ名
    key: str         # ホットキー
    ctrl/shift/alt   # 修飾キー
    pmis: Collection # アイテムのリスト
    data: str        # URL形式の追加設定
    poll_cmd: str    # 表示条件

PMIItem (アイテム定義)

class PMIItem(PropertyGroup):
    name: str      # 表示名
    mode: str      # COMMAND, PROP, MENU...
    text: str      # コマンド/プロパティパス
    icon: str      # アイコン名 + フラグ
    enabled: bool  # 有効/無効

PMLink (ツリービュー用)

エディターUIのツリー表示のためのリンク構造。フォルダ階層を表現。

ポイント: PMItemの mode プロパティが 「このデータをどのエディターが解釈するか」を決定する。
mode = 'PMENU' → ed_pie_menu.py が処理
mode = 'MACRO' → ed_macro.py が処理
Slide 5

データエンコーディング

URL形式の設定保存パターン

PMItem.data の構造

追加設定は URL クエリパラメータ形式で格納されます:

# 例: パイメニューの追加設定
"pm?pm_radius=150&pm_threshold=20&pm_flick=True"

# 例: ポップアップダイアログの設定
"pd?pd_title=True&pd_box=False&pd_width=300"

解析の仕組み (pme.py)

class ParsedData:
    def __init__(self, text):
        self.type, _, data = text.partition("?")  # "pm" と残り

        for prop in data.split("&"):              # 各パラメータ
            k, v = prop.split("=")
            setattr(self, k, decode(v))           # 動的に属性設定

# 使用例
prop = pme.props.parse(pm.data)
radius = prop.pm_radius  # 150
脆弱性ポイント:
この形式は柔軟だが、以下の問題がある:
  • 型安全性がない (全て文字列から変換)
  • スキーマが明示されていない (props_mapで管理)
  • バージョン間の互換性維持が手動
Slide 6

アイテムモード (EMODE_ITEMS)

PMIItemが取りうる振る舞い

メニューの各アイテムは mode によって実行時の振る舞いが決まります:

Mode 説明 textフィールドの内容
COMMAND Pythonコード実行 bpy.ops.mesh.primitive_cube_add()
PROP プロパティウィジェット表示 C.scene.render.engine
MENU サブメニュー/PMEメニュー参照 MySubMenu or @MySubMenu
HOTKEY ホットキーに紐づくオペレーター実行 Ctrl+Z
CUSTOM カスタムレイアウトコード L.prop(C.object, "name")
EMPTY セパレーター/空スロット "" or "column"
@フラグ (F_EXPAND):
@MenuName の形式は「展開して表示」を意味する。
サブメニューをボタンではなく、その場に展開表示する。

Slide 8

PMEContext - 実行時コンテキスト

pme.py の心臓部

PMEContextの役割

メニュー描画・コマンド実行時に必要な情報を保持するシングルトン。
pme.context としてグローバルにアクセス可能。 

class PMEContext:
    def __init__(self):
        self._globals = dict(
            bpy=bpy,
            D=bpy.data,      # ショートカット
            T=bpy.types,
            O=bpy.ops,
            P=bpy.props,
            # ... 便利な変数群
        )

        # 現在処理中のメニュー/アイテム
        self.pm = None       # 現在のPMItem
        self.pmi = None      # 現在のPMIItem
        self.index = None    # アイテムインデックス

        # 描画時の一時情報
        self._layout = None  # UILayout (L でアクセス)
        self._event = None   # Event (E でアクセス)

    def gen_globals(self):
        """exec/eval用のグローバル辞書を生成"""
        return dict(
            text=self.text,
            icon=self.icon,
            PME=temp_prefs(),
            PREFS=get_prefs(),
            **self.globals
        )
設計意図:
ユーザーが書くコマンド内で L.prop(...) や C.object の ような短縮形を使えるようにするため。
これによりPMEのコマンドは非常に簡潔に書ける。

エディター登録の仕組み

Editor()関数パターン

各エディターモジュールのregister()

# ed_pie_menu.py の例
class Editor(EditorBase):
    id = 'PMENU'
    icon = 'NONE'
    default_name = "Pie Menu"
    has_hotkey = True
    supported_slot_modes = {'COMMAND', 'PROP', 'MENU', ...}

    def on_pm_add(self, pm):
        # パイメニューは常に10個のスロットを持つ
        for i in range(10):
            pm.pmis.add()

    def draw_items(self, layout, pm):
        # パイメニュー用のスロット配置UI

def register():
    Editor()  # ← これで PMEPreferences.editors に登録される

preferences.py での利用

class PMEPreferences(AddonPreferences):
    editors = {}  # mode → Editor インスタンス

    def ed(self, mode):
        """mode からエディターを取得"""
        return self.editors.get(mode)

    def add_pm(self, mode, name=None):
        ed = self.ed(mode)
        pm = self.pie_menus.add()
        pm.mode = mode
        pm.name = name or ed.default_name
        ed.on_pm_add(pm)  # エディター固有の初期化
        return pm
この設計の利点:
新しいメニュータイプを追加する場合、ed_*.py を作成して MODULES に追加するだけで良い。プラグイン的な拡張性。

ホットキー管理

keymap_helper.py の仕組み

ホットキー登録の流れ

# PMItem.register_hotkey() の概要
def register_hotkey(self, km_names=None):
    pr = get_prefs()

    for km_name in km_names:
        pr.kh.keymap(km_name)  # キーマップを選択
        kmi = pr.kh.operator(
            WM_OT_pme_user_pie_menu_call,
            None,
            self.key,           # 'A', 'SPACE', etc.
            self.ctrl,
            self.shift,
            self.alt,
            self.oskey,
            self.key_mod,
            self.any,
        )
        kmi.properties.pie_menu_name = self.name

        # 登録状態を記録
        self.kmis_map[self.name] = {km_name: kmi}
脆弱性ポイント: キーマップ待機問題
Blender起動時、全てのキーマップがすぐに利用可能ではない。
PME_OT_wait_keymaps オペレーターでタイマー監視して、 キーマップが利用可能になったら登録を試みる。
これが missing_kms の管理につながっている。

セッション復元

__init__.py のハンドラー

問題: Blenderファイル読み込み時のプリファレンス消失

Blenderは新しいファイルを開くとき、一時的にAddonPreferencesが リセットされることがある。PMEはこれに対処する必要がある。 

# load_pre_handler: ファイル読み込み前
@persistent
def load_pre_handler(_):
    global tmp_data
    # 現在の設定を辞書形式でバックアップ
    tmp_data = property_utils.to_dict(get_prefs())

# load_post_handler: ファイル読み込み後
@persistent
def load_post_handler(filepath):
    global tmp_data
    if not bpy.data.filepath:  # 新規ファイルの場合
        # バックアップから復元
        property_utils.from_dict(get_prefs(), tmp_data)
    tmp_data = None
Blender 5.0以降の変更:
このコードはBlender 5.0未満でのみ動作。5.0以降はプリファレンスの 永続化方式が変わり、このハックは不要になった（はず）。
ただし、完全に検証されていない脆弱性ポイント。

動的クラス生成

メニュー・パネルの動的登録

通常メニュー (RMENU) の例

# ui_utils.py
def get_pme_menu_class(pm_name):
    """PMEメニュー用のMenuクラスを動的生成"""
    class_name = "PME_MT_" + re.sub(r'\W', '_', pm_name)

    if class_name in pme_menu_classes:
        return class_name

    # 動的にクラスを生成
    cls = type(
        class_name,
        (bpy.types.Menu,),
        {
            'bl_label': pm_name,
            'bl_idname': class_name,
            'draw': gen_draw_func(pm_name),  # draw関数も動的生成
        }
    )

    bpy.utils.register_class(cls)
    pme_menu_classes[class_name] = cls
    return class_name

パネル拡張 (既存パネルへの追加)

# ed_base.py - extend_panel()
def extend_panel(pm):
    tp = getattr(bpy.types, panel_name, None)
    if tp and issubclass(tp, bpy.types.Panel):
        draw_func = gen_panel_draw(pm.name)
        tp.append(draw_func)  # 既存パネルに描画関数を追加
        EXTENDED_PANELS[pm.name] = draw_func
設計意図:
Blender APIでは Panel/Menu クラスを事前に定義する必要があるが、 PMEはユーザーが任意の数のメニューを作れる。
この矛盾を解決するため、クラスを実行時に動的生成している。

脆弱性ポイント一覧

注意が必要な領域

領域 脆弱性 影響
セッション復元 to_dict/from_dict によるデータドリフト 設定消失、データ破損
キーマップ登録 タイミング依存、missing_kms管理 ホットキー動作せず
URL形式data スキーマなし、型安全性なし パース失敗、デフォルト適用漏れ
動的クラス生成 名前衝突、アンレジスト漏れ Blenderクラッシュ
exec/eval ユーザーコードの例外処理 UIフリーズ、エラー表示
Poll関数 動的コンパイル、キャッシュ管理 パフォーマンス、メモリリーク
最も危険な箇所:
PMItem.poll_methods - poll関数をコンパイルしてキャッシュするが、 メニュー削除時にクリーンアップされない可能性がある。
Slide 16

暗黙の設計哲学

オリジナル作者から継承されたもの

1. 「とにかく動かす」優先

型チェックや厳密なバリデーションより、柔軟性と即時動作を優先。
try/except で囲んで print_exc() するパターンが多い。

2. グローバル状態の活用

pme.context, lh (LayoutHelper), クラス変数など、 グローバルにアクセス可能な状態を多用。これは短いコードを書けるが、 テスタビリティを下げる。

3. Blender APIへの密結合

抽象化レイヤーなしでBlender APIを直接使用。
これはパフォーマンスには良いが、Blenderバージョン変更に脆弱。

4. 「一つのデータ構造で全て表現」

PMItem/PMIItem で10種類以上の異なる概念を表現。
シンプルだが、各タイプ固有の制約を表現しにくい。

これらは「悪い設計」ではない:
単独開発者が機能を素早く追加するには合理的。
ただし、チーム開発やメンテナンスフェーズでは課題になる。

モジュール依存関係

MODULES順序の意味

インポート順序 (依存方向は下→上) ═══════════════════════════════════════════════════════════════════ [Core Layer] addon ─────────────────────── 基本設定、get_prefs(), temp_prefs() │ pme ───────────────────────── PMEContext, props (データパーサー) │ constants ─────────────────── 定数、ENUMアイテム [Utility Layer] utils, bl_utils ───────────── 汎用ユーティリティ │ layout_helper ─────────────── lh シングルトン (UI構築ヘルパー) │ keymap_helper ─────────────── KeymapHelper, ホットキー管理 [Data Layer] types ─────────────────────── PMItem, PMIItem, Tag, PMLink │ operators ─────────────────── 主要オペレーター群 [Editor Layer] ed_base ───────────────────── EditorBase 基底クラス │ ed_pie_menu, ed_menu, ... ── 各エディター実装 [UI Layer] preferences ───────────────── PMEPreferences, 設定UI

この順序は循環インポートを避けるために重要。
例: operators.py は types.py より後なので、 types.py で operators を import できない。
Slide 18

PME2への提言

保持すべきもの PRESERVE

1. 統一データモデルのコンセプト

PMItem/PMIItem の「全てはメニューアイテム」という抽象化は強力。
ただし、型レベルでの制約は追加すべき。

2. PMEContextによるグローバル変数提供

L, C, D などの短縮形はUXとして優秀。
ユーザーコードの簡潔さを維持すべき。

3. EditorBaseパターン

各メニュータイプごとにエディターを分離する設計は良い。
プロトコル(インターフェース)として明示化するとさらに良い。

4. JSON形式のエクスポート/インポート

ポータビリティの確保は重要。
スキーマバージョン管理を追加すべき。

5. LayoutHelper (lh) のアプローチ

Blender UILayoutのラッパーは便利。
ただし、より型安全にすべき。

Slide 19

PME2への提言

再設計すべきもの REDESIGN

1. データエンコーディング

現在: 
"pm?pm_radius=150&..."
提案: 
@dataclass
class PieMenuData:
    radius: int = 150
    threshold: int = 20

2. 状態管理

グローバルシングルトンを減らし、明示的な依存性注入へ。
特に PMItem.kmis_map のようなクラス変数は危険。

3. セッション復元

to_dict/from_dict パターンは脆弱。
Blender 5.0+の新しいプリファレンス永続化を活用すべき。

4. エラーハンドリング

try/except/print_exc() の乱用を減らす。
Result型やOptional型で明示的なエラー伝播を。

5. Poll関数のコンパイル

動的な compile() は避け、
宣言的なコンテキスト条件システムを検討。

Slide 20

PME2への提言

削除を検討すべきもの REMOVE

1. c_utils.py (Cython/ctypes操作)

Blender内部構造体へのアクセスは非常に脆弱。
Blenderバージョンごとにオフセットが変わる可能性がある。

2. 過度なレガシー互換性コード

compatibility_fixes.py の古いバージョン対応。
PME2では最低サポートバージョンを上げ、クリーンスタート。

3. 複雑なセッション復元ロジック

Blender 5.0未満のハックは不要になる可能性が高い。

4. 使用頻度の低い機能

ユーザー調査を行い、ほとんど使われていない機能を特定して削除検討。
例: 特殊なモーダル操作、複雑なマクロ機能など。

注意: 削除は慎重に。
既存ユーザーの設定との互換性を考慮したマイグレーションパスが必要。

まとめ

PMEアーキテクチャ理解のキーポイント

データ構造

  • PMItem = メニュー定義
  • PMIItem = アイテム定義
  • mode で振る舞いを決定
  • URL形式のdata拡張

実行フロー

  • Keymap → Operator → Draw → Exec
  • pme.context で状態共有
  • 動的クラス生成でUI登録

設計パターン

  • EditorBase継承パターン
  • グローバルシングルトン多用
  • try/except防御的プログラミング

脆弱性

  • セッション復元
  • キーマップタイミング
  • 型安全性の欠如
最終メッセージ:
PMEは「動くことが正義」の時代に作られた素晴らしいツール。
PME2ではその精神を継承しつつ、モダンな設計原則を取り入れることで、
より保守しやすく、拡張しやすく、テストしやすいコードベースを目指そう。
